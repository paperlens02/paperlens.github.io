基础战斗机制设计

发表于 2022-07-12 更新于 2022-07-20 分类于 paperRPG ， 文档 ， 撰写中

一、设计思路

基于回合制，实现轻量级的，快速的战斗体验。由于大地图探索机制中”允许不触发任何战斗”的设计，需要移除战斗体验与属性之间的一些绑定；故而需要为战斗设计新的属性，新的属性应与大地图体验相关。

二、设计目标

  1. 设计基于回合制的基础战斗规则。
  2. 初步设计战斗属性和伤害计算
  3. （可选）提升战斗的实时策略感

三、详细设计_机制篇

3.1 回合制战斗规则

四、详细设计_设计篇

4.1 一些基础原则和设计目标

  • 在游戏基础设计中，为满足多周目体验以及提升解谜专注度，设计战斗为可规避玩法。因此不能将战斗与属性成长硬关联；但又需要为战斗玩法提供足够的奖励。为此引出以下设计：
    • 将战斗与地图探索系统关联。
      • 战斗胜利后，可以对地图造成某种影响。如：降低地图中所有怪物移速、降低怪物密度；或是显示隐藏交互物等
    • 将战斗与剧情系统相关联
      • 战斗结果会影响”灵视值”，对剧情产生影响。如：增加对话选项、识破谎言等
    • 为战斗设计增益其他系统的属性/道具奖励
      • 上述”灵视值”其实就是其中的一种：拥有一定的灵视值后，玩家可以更轻松的解谜、看到更清晰的剧情，并以此在其他游戏系统中抄近路。这符合多周目设计的思路：在一周目维持游戏难度，但给出提示；在多周目允许玩家直接像开透一样玩游戏。这个设计并不是为不同玩家提供不同游戏体验，而是为同一玩家提供更具阶梯性的多周目体验。
      • 道具类
        • 提供加速道具、冻结道具等等策略性道具。在具体怪物上有具体掉落，为玩家提供清晰的掉落提示。这本质上是为大地图探索增加策略性：玩家可以主动选择接战来修正自己的大地图探索体验。
        • 提供货币类道具。这是一种奖励保底设计。暂不考虑”仅掉落货币的战斗”。暂时认为即使是战斗失败，也应获得货币。
      • 属性类：
        • 回复属性惩罚。在部分系统中，会存在惩罚性设计。可以通过战斗回复这些惩罚。
        • todo
    • 战斗机制本身的成长性设计
      • 综述：由于不能将战斗与属性成长硬关联，也就不存在等级之类的战斗属性，战斗也不能提供升级之类的效果。但我们仍需要随着游戏进程提高战斗的维度。因此在设计上提供随游戏进程线性成长的战斗内容。
      • 策略方面（配队、技能、道具）：
        • 随着游戏推进让角色入队。设计各个角色之间的配合，以及战斗与配队间的策略关系，从而提升后期战斗维度。
        • 随着游戏推进让玩家购买/解锁更强大的技能。强大的技能与剧情推进绑定，并作为一些解谜奖励出现。在后期，玩家可以通过更高级的技能来应对高维度的战斗。
        • 随着游戏推进购买/收集更多的道具（这些道具不来自于战斗系统）。在战斗中提高道具使用的存在感。（我们剧情设定上更多鼓励智取）
      • 属性方面：
        • todo: 可否不与游戏进程绑定，而是与玩家熟练度绑定？
      • 注：
        • 线性成长的是战斗内容，而非战斗强度。按照”仅用默认角色和默认技能即可通关”的目标进行难度设计。策略提供的是更广的战斗内容、体验的扩充。
    • 提升战斗玩法本身的存在意义
      • 综述：虽然可以完全不战斗通关，但如果进行一定程度的战斗，可以获得某些奖励的宏观设计。

    • 策略方面：
      • 设计克制关系。

      • 设计连击。
        • 设计连击的目的：在玩家熟练度高的状况下，最大限度的收缩战斗耗时；提升正确操作的正向反馈；配合画面和声音表现提升爽度。

buff表-效果类型

  • 新增14-每次被编辑A属性时额外编辑B属性（类似9，特殊效果）
    • 效果参数配置格式为”A属性名,A属性编辑值的要求,B属性名(精确到_abs),B属性编辑值(支持配置为公式)”
    • 与效果类型9的差别: 9为”主动编辑他人属性”时，即在实际属性编辑之前就将属性id转化掉了；14为”收到属性编辑”时，即在实际属性编辑完成后，再根据编辑结果额外添加内容
      • 例1：实际被编辑的是max_health，但因为同时影响了health属性，同样会满足条件
      • 例2：实际编辑health属性时因为有护盾，最终没能影响health属性，不满足条件
  • 新增15-每次调用属性时按条件加成（特殊效果）
    • 效果参数配置格式为”属性名,条件,加成公式”
  • 新增16-受到伤害时获得新buff（buff时机id6，特殊效果）
    • 效果参数配置格式为”buffid,buff概率,buff层数”，暂不支持用|分隔配置多个
  • 新增17-堆叠n次后触发属性加成（类似12，特殊效果）
    • 效果参数配置格式为”触发次数，属性名，加成值”
  • 新增18-行动开始时受到物理伤害（buff时机id1，一次性效果表id5）
  • 新增19-回合结束时受到自然伤害（buff时机id4，一次性效果表id13）

  • 新增5-为友方回复后
  • 新增6-自身护盾消失后

  • 新增8-触发弱点效果后
  • 新增9-击杀敌方单位后
  • 新增10-自身受到伤害后

实际配置

  • 每次释放技能时，攻击力提升5%，最多可以委加5层, 持续整场战斗
    • 配置触发类技能，触发时机为4-自身释放技能后，Action效果为添加Buff，Buff配堆叠上限
    • 注：单位死亡等特殊情况这个buff会被清理掉
  • 为受到自身治疗的单位提升20%基础攻击力，持续2回合，不可叠加
    • 配置触发类技能，触发时机为5-为友方回复后，Action目标范围为15.触发技能的触发源中的目标，Action效果为添加Buff
    • 需要注意的是这属于触发技能，需要依照技能排序；若实际触发技能执行时找不到原目标，则跳过
  • 自身护盾消失时立刻回复1点bp
    • 配置触发类技能，触发时机为5-自身护盾消失后，配置触发类技能即可
  • 在对敌方单位造成伤害时，有20%的概率降低该单位5%的魔法防御力，持续2回合(效果可叠加)
    • 配置触发类技能，触发时机为6-对敌方单位造成伤害后，Action目标范围为15.触发技能的触发源中的目标
  • 每次触发弱点效果后，提高自身10%的攻击力，持续2回合(效果可叠加)
    • 配置触发类技能，触发时机为7-触发弱点效果后
    • 需要注意的是，每一个施法者（即配置中的a）被判定为为该单位的元素反应技能都会执行触发。
  • 每次释放技能后，会返还该技能所消耗的BP的1/4
    • 配置触发类技能，触发时机为4-自身释放技能后
  • 自身攻击造成敌方单位死亡后，提升自身15%的攻击力，持续2回合
    • 配置触发类技能，触发时机为8-击杀敌方单位后
    • 只要造成死亡的效果中的施法者（即配置中的a）为自身，就会触发
  • 每次释放技能后, 对敌方生命值最低的单位造成攻击力90%的自然元素伤害
    • 配置触发类技能，触发时机为4-自身释放技能后
  • 在自身受到伤害时，为自己回复攻击力40%的生命值
    • 配置被动技能，为自身上永久buff
    • buff效果类型为14-每次被编辑A属性时额外编辑B属性 效果参数为”health,value<0,health_abs,b.attack*0.4”
  • 当自身生命值高于70%时，攻击力提高20%
    • 配置被动技能，为自身上永久buff
    • buff效果类型为15-每次调用属性时按条件加成 效果参数为”attack,b.health/b.max_health>0.7,”
  • 每次受到伤害时，积攒1层怒气，怒气到达5层时，清空所有怒气，并且伤害提高100% (伤害提高100%的效果持续2回合)
    • 配置被动技能，为自身上永久buff
    • buff效果类型为16-受到伤害时获得新buff，效果参数如”怒气buffid,1,1”
    • 配置怒气buff，怒气buff效果类型为17-堆叠n次后触发属性加成，效果参数如”5,damage_abs,1”

基于回合战棋的大地图探索和解谜机制设计

发表于 2022-07-07 更新于 2022-07-12 分类于 paperRPG ， 文档 ， 撰写中

一、设计思路

世界地图设计承载了剧情表现力和主要玩法，是游戏各个细分玩法的入口。本游戏世界地图需遵循小而精的设计原则，在尽可能小的线性地图中做足量交互填充和玩法设计。为进一步加强设计感、提升地图交互的可玩性，本作参考回合制战棋玩法的思路，在不降低自由探索感的前提下进行大地图交互和解谜设计。

二、设计目标

  1. 在世界地图中，基于交互物和事件的填充，设计与剧情、养成等系统紧密关联的解谜内容。
  2. 在世界地图中填充自主巡逻的怪物，接入战斗系统，提升策略性。
  3. 设计弱回合制的概念。回合与移动步数相关联，通过回合的进展触发世界地图中各项事件（包括怪物巡逻、属性变化、表现变化）。

三、详细设计_机制篇

3.1 地图基本构成

  • 地图是由多个格子构成的。

    • 格子是构成地图的最小单位；格子是构成地图的唯一元素。
    • 每个格子上可以放置多个事件、交互物等。
    • 所有单位必须停留在某个格子上；单位移动必然跨过整个格子；存在同时占有多个整数格子的单位。
    • 从表现上来讲，在整个游戏中所有的地图的格子，其客观大小和意义是一致的。

  • 地图表现与格子无关。

    • 为提升表现力，地图表现为一个完整的场景，先出场景后设计格子。

3.1.1 格子

  • 格子拥有属性：
    • 可否移动

3.2 地图填充

3.2.1 交互物

  • 交互物的交互位置被安置在一个或多个地图格子上。
  • 交互物的表现可以是动态加载的单个模型，也可以是绑定在地图场景上的某个物体。

3.2.2 地图事件

  • 地图事件与格子绑定，在满足触发条件时触发。详见[事件机制设计]。

3.2.3 怪物

  • 怪物是依据配置或程序逻辑被动态生成在地图上的。
  • 怪物拥有自身的AI，例如自主巡逻AI。详见[怪物AI设计_大地图篇]。
  • 怪物拥有自身的事件触发器，例如接战事件。详见[怪物事件设计_大地图篇]。

3.3 地图中的玩家

  • 暂时不考虑组队系统。玩家仅主角一人。

3.3.1 出生点

  • 各个地图需要配置多个出生点，对应各个进入地图来源。地图初始化完毕后，玩家被生成在出生点上。

3.3.2 玩家的移动

  • 玩家的移动是以格子为单位的。
  • 在自由活动时间，玩家可以任意操作移动

3.4 回合

  • 回合数会随着玩家的移动自增。
  • 基本设计是玩家每移动n步，回合数+1。这个n的值在游戏大部分时候是固定的某个值，它不受玩家属性影响，但可能受地图效果影响。

3.4.1 回合与地图表现

  • 地图表现会随着回合推进而改变。（细节待设计）

3.4.2 回合与地图填充物

  • 交互物、事件均有可能随着回合推进而改变。（细节待设计）
  • 怪物会依据回合推进执行AI。

3.4.3 回合与玩家属性

  • （细节待设计）

四、详细设计_设计篇

4.1 一些基础原则和设计目标

  • 所有战斗均可以规避。
  • 减少flag强依赖。交互物可调查，但无需完全调查，允许通过直觉或猜测(枚举)解谜。

4.2 交互物的设计

  • 所有交互物均有其交互意义。即：不设计类似*调查椅子->”这是椅子，坐起来很舒服”*这种调查物件。
    • 交互意义：即必须与某游戏系统有关。包括但不限于：
      • 提供事件flag或解谜线索（与探索解谜系统相关）
      • 提供剧情内容（与剧情系统相关）
      • 提供属性、buff等等（与战斗/养成系统相关）
      • …..
  • 在每个场景中的交互物数量、比例、效果程度应有一定规律
    • 不应存在很空/很挤的场景；不应存在不同场景中某种交互物效果差异过大。
      • 如：在A场景中的所有交互物共可提供属性值x点，在紧接着的B场景中总属性值也应接近x点。
      • 从游戏前期到后期，交互物整体密度会逐步提升，但不可时多时少。
      • 在各个场景中，与每个系统相关的交互物所占的比例应尽可能稳定。
  • 预计会设计隐藏交互物。按一定密度(隐藏交互物/场景)配置。

4.3 怪物的设计

4.4 解谜的设计

  • 解谜需服务于某其他系统。即：谜题的谜底指向某剧情细节/解谜的奖励带来某养成系统数值
  • 解谜完成后有某种形式的奖励。这种奖励需要是长久的，不能是临时的。不一定是显式的。
  • 解谜可以设置分支答案。但依旧不依赖flag。
  • 在每个场景中的解谜数量、比例、难度应有一定规律（同上4.2）

saturate Saturate is a function that clamps it’s input between 0 and 1.
normalize float normalize(float x)
vec2 normalize(vec2 x)
vec3 normalize(vec3 x)
vec4 normalize(vec4 x)
normalize() returns a vector with the same direction as its parameter, x, but with length 1.
dot 点乘。
float dot(float x, float y)
float dot(vec2 x, vec2 y)
float dot(vec3 x, vec3 y)
float dot(vec4 x, vec4 y)
dot() returns the dot product of two vectors, x and y. i.e., x[0]⋅y[0]+x[1]⋅y[1]+… If x and y are the same the square root of the dot product is equivalent to the length of the vector. The input parameters can be floating scalars or float vectors. In case of floating scalars the dot function is trivial and returns the product of x and y.
